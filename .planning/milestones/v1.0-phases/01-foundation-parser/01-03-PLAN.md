Cache loaded languages and compiled queries — each grammar loads once per process.

  <action>
Create `src/parser/languages/typescript.ts` — extracts CodeNode objects from TS/JS/TSX source files.

Uses `tags.scm` query captures from the grammar plus custom patterns for constructs not covered by default tags.

Export:
```typescript
export async function extractTypeScriptNodes(
  filePath: string,
  source: string,
  languageId: LanguageId
): Promise<readonly CodeNode[]>
```

Extraction strategy:
1. Parse the source using `parseSource()` from tree-sitter.ts
2. Run `tags.scm` query captures against the tree — this gives `@definition.function`, `@definition.class`, `@definition.method`, `@reference.call`, etc.
3. For each capture, construct a CodeNode with:
   - `id`: `${filePath}:${name}:${startLine}`
   - `name`: from the `@name` capture associated with the definition
   - `kind`: map capture name to NodeKind (definition.function → 'function', definition.class → 'class', etc.)
   - `filePath`: absolute path
   - `startLine`/`endLine`: from the capture node's position
   - `references`: collect `@reference.*` captures from within the definition scope

4. Additional extraction beyond tags.scm (handle PARSE-04 requirements):
   - **Type aliases**: Walk AST for `type_alias_declaration` nodes → kind 'type'
   - **Enums**: Walk AST for `enum_declaration` nodes → kind 'enum'
   - **Interfaces**: Should be in tags.scm, but verify — fall back to AST walk for `interface_declaration`
   - **Barrel re-exports**: Walk for `export_statement` nodes that contain `export { ... } from '...'` or `export * from '...'` → kind 'export' with references pointing to the source module
   - **Decorators**: Walk for `decorator` nodes preceding class/method declarations → include decorator name in the parent node's references
   - **Import statements**: Walk for `import_statement` and `import_declaration` → kind 'import' with references to imported names

5. Deduplicate nodes by id (tags.scm and AST walk may produce overlaps)
6. Sort nodes by startLine

Extraction strategy (same pattern as TypeScript extractor):

   - **Relative imports**: Walk for `import_from_statement` nodes where the module starts with `.` → kind 'import', references include the imported names, and the module path includes the relative prefix
   - **`__init__.py` re-exports**: Walk for `import_from_statement` nodes that import from relative paths → kind 'export' when in `__init__.py` files (these act as barrel files)
   - **Constants**: Walk for top-level `assignment` nodes where the target is UPPERCASE → kind 'constant'

5. Deduplicate and sort by startLine

Create `src/parser/index.ts` — the parser orchestrator that dispatches to language-specific extractors.

Exports:
```typescript
// Parse a single file and return its CodeNode results
export async function parseFile(filePath: string): Promise<ParseResult>

// Parse multiple files concurrently and return all results
export async function parseFiles(filePaths: readonly string[]): Promise<readonly ParseResult[]>
```

Logic for `parseFile`:
1. Read file contents from disk (`fs.readFile`)
2. Determine language from extension using `fileExtensionToLanguage`
3. If unsupported language, return empty ParseResult (don't error — graceful skip)
4. Ensure parser is initialized (call `initParser()` if first use — cache the init promise)
5. Dispatch to the appropriate extractor:
   - 'typescript' | 'tsx' | 'javascript' → `extractTypeScriptNodes(filePath, source, languageId)`
   - 'python' → `extractPythonNodes(filePath, source)`
6. Measure parse time using `performance.now()`
7. Return `ParseResult` with filePath, nodes, and parseTimeMs
8. On parse error: throw `ParseError` with actionable hint including the file path and language

Logic for `parseFiles`:
1. Process files with bounded concurrency (max 8 concurrent parses to avoid memory spikes)
2. Collect all ParseResults
3. Log summary via `logger.info`: total files, total nodes, total time
4. Return results sorted by filePath

Use a simple concurrency limiter (Promise pool pattern) — no external dependency needed.
