MCP SDK tool schema access:
The McpServer instance does not expose a public method to list registered tool schemas directly.
To test schema size, either:
(a) Create a McpServer, register tools, then use the SDK's internal listTools handler
(b) Manually construct the Zod schemas and measure their JSON Schema output via z.object().shape
(c) Import the handler functions and measure the Zod-to-JSON-Schema conversion

Approach (a) is most accurate: create a McpServer, register all tools, then call the listTools handler.
The MCP SDK's Client class can be used in tests to connect to the server and call listTools.

Create `tests/mcp/schema-size.test.ts`.

The most reliable approach: use the MCP SDK's Client + InMemoryTransport to connect to the server and call listTools. This gives the exact JSON Schema that MCP clients see.

```typescript
import { McpServer } from '@modelcontextprotocol/sdk/server/mcp.js';
import { Client } from '@modelcontextprotocol/sdk/client/index.js';
import { InMemoryTransport } from '@modelcontextprotocol/sdk/inMemory.js';
```

If InMemoryTransport is not available in the CJS build, fall back to approach (b): extract the Zod schemas from the tool registration and measure their JSON Schema conversion using `zodToJsonSchema` or a manual `z.object(shape).describe(desc)` serialization.

Alternative simpler approach if SDK internals are awkward: Import the tool registration function from server.ts, create a minimal McpServer, register tools, and use `server.server.listTools()` (the low-level server instance) to get schemas. Measure `JSON.stringify(result.tools).length`.

The test should:
1. Register all 5 tools on a fresh McpServer (extract tool registration to a `registerTools(server, rootDir)` function in server.ts if not already done)
2. Get the tool list via the server's internal handler
3. Serialize all tools to JSON
4. Assert total length < 4000

If direct server access is too complex, the simpler fallback: manually define the same Zod schemas used in server.ts, convert to JSON Schema via `zodToJsonSchema` (from zod-to-json-schema, already a transitive dep of MCP SDK), and measure.

The test name should be: "total MCP tool schema serialization stays under 4000 characters"

   - stderr contains spinner output and/or logging
   - stdout does NOT contain `[graft:` log prefix

4. **serve command does NOT write non-JSON-RPC to stdout:** This is harder to test in a unit test since serve blocks on stdin. Approach: spawn the process, send nothing to stdin, wait 500ms, kill it, verify stdout was empty (no banner, no spinner, no "MCP server started" message leaked to stdout). Only stderr should have startup logging.

Use the same test fixture directory pattern as Phase 1 integration tests. The `tests/fixtures/` directory should have enough files for discovery + parsing to work.

CRITICAL: The build must be run before these tests work (`pnpm build` produces `dist/index.cjs`). If the existing test does this via a beforeAll, follow the same pattern.

Note: chalk auto-detects TTY. In test (non-TTY), chalk output is plain text (no ANSI codes). This is fine -- the test is about stream separation (stdout vs stderr), not about whether colors render.
